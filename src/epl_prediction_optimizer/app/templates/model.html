Two-Stage Architecture

Stage A and Stage B are deliberately separate — improving one does not require changing the other.

Stage A · Probability Model

HistGradientBoosting trained on 30 seasons. Outputs P(home win), P(draw), P(away win) for every fixture. Optimized for log-loss calibration, not contest constraints.

Features: Elo diff, rolling form, home/away form, win streak, in-season table position.

→

Stage B · Pick Optimizer

Integer programming (PuLP). Takes Stage A probabilities as input. Applies contest constraints: one pick per week, each team 1–2 uses, each venue at most once.

Maximizes expected total season points subject to constraints.

Reusable Analysis Scripts

Run these from the repo root — they're permanent, not one-off commands.

# Feature importance + training metrics + backtest comparison
uv run scripts/analyze_model.py
uv run scripts/analyze_model.py --season 2425
uv run scripts/analyze_model.py --no-importance

# Run and compare backtests
uv run scripts/compare_backtests.py
uv run scripts/compare_backtests.py --run 2324 2425
uv run scripts/compare_backtests.py --picks 2425

# CLI equivalents
eplpo analyze --season 2425
eplpo backtest --season 2324
eplpo train && eplpo predict && eplpo optimize
